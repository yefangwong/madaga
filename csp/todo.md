    - 安裝 Ollama 並拉取建議模型。
    - 微調 `Synthesizer.java` 的 Prompt 以更精確引導模型輸出 SQLite 兼容語法。

主題：建構整合 Text-to-SQL 與 CVE 修復功能的智能查詢平台之綜合戰略規劃

1. 理解目標


我們的綜合目標是打造一個雙核心功能的企業級智能平台。核心一 (Text-to-SQL)：讓用戶透過自然語言與數據庫互動，生成並驗證 SQL 查詢。核心二 (CVE
修復)：自動化發現並協助修復程式碼中的安全漏洞。此平台旨在同時提升數據可及性與程式碼安全性，將開發與維運的兩大痛點整合到一個統一的使用者介面
中。

2. 調查與分析

在啟動前，我們必須對兩個功能領域的共性和差異性進行分析，以確保資源的有效利用。


* 共性分析：
    * 統一的後端框架：兩個功能都將基於現有的 Java/Spring Boot 架構。
    * 統一的前端介面：所有功能最終都將整合到同一個前端 UI 中，必須考慮整體的版面配置與使用者體驗。
    * 非同步任務處理：無論是執行 SQL 查詢還是應用 CVE 修復，都可能是耗時操作，需要統一的非同步任務處理機制 (如任務隊列、WebSocket
      狀態更新)。


* 差異性與依賴分析：
    * Text-to-SQL 更側重於數據庫互動、自然語言處理 (NLP) 模型整合和查詢結果的視覺化。
    * CVE 修復 更側重於與外部程式碼託管服務 (如 GitHub API) 的整合、檔案系統的讀寫操作以及程式碼依賴的解析。


* 開工前必須回答的關鍵問題：
    1. UI/UX 整合策略：這兩個功能在 UI 上應該是並列的獨立功能區，還是在特定情境下（例如分析與程式碼相關的數據時）進行聯動？
    2. 身分驗證與權限模型：用戶是否有權限同時存取數據庫和程式碼庫？需要設計一個能同時管理這兩種資源權限的統一模型。
    3. 後端服務的邊界：是將兩個功能的所有服務都放在同一個 Spring Boot 應用中，還是根據功能（數據庫服務 vs. 程式碼服務）拆分為不同的微服務？

3. 戰略執行步驟 (含優先順序)


以下是整合後的執行步驟，已按優先順序標記。我們將採用迭代方式，先專注於實現每個功能的核心價值 (MVP)，再進行擴展。


* 階段一：核心基礎建設與 Text-to-SQL MVP
    1. 🟥 後端基礎建設：統一配置與 API 骨架
        * 任務: 搭建 Spring Boot 應用，配置好數據庫連接、安全框架 (Spring Security)、日誌系統 (Logback)，並為兩大功能定義好初步的 REST API
          路由結構。
        * 產出: 一個可運行的 Spring Boot 應用骨架，包含 /api/text-to-sql/... 和 /api/vulnerabilities/... 的空 API 端點。
    2. 🟥 後端 Text-to-SQL：Schema 萃取服務
        * 任務: 實現 SchemaService，能連接數據庫並提取元數據。
        * 產出: GET /api/schema/{dbId} API，能返回數據庫結構的 JSON。
    3. 🟥 前端建設：搭建 UI 框架與數據庫結構樹渲染
        * 任務: 建立前端應用的基本佈局，包括左側邊欄，並實現 Schema 瀏覽器組件，能呼叫後端 API 並渲染數據庫結構樹。
        * 產出: 一個具備基本佈局且能清晰顯示數據庫 Schema 的前端介面。
    4. 🟧 後端 Text-to-SQL：核心轉換服務 (Mock)
        * 任務: 實現 TextToSqlService，接收問題並返回一個寫死的、預設的 (mocked) SQL 查詢結果和解釋。暫不整合真實的 AI 模型。
        * 產出: POST /api/sql/generate API，能穩定返回一個範例 SQL。
    5. 🟧 前端 Text-to-SQL：核心互動流程
        * 任務: 實現主內容區的輸入框與結果展示區，讓用戶可以輸入問題，點擊生成，並在前端看到從後端 mock API 返回的 SQL 和解釋。
        * 產出: Text-to-SQL 功能的核心使用者流程 (從輸入到展示) 可用。


* 階段二：功能深化與 CVE 修復 MVP
    1. 🟥 後端 CVE：弱點數據獲取服務
        * 任務: 實現 VulnerabilityService，能從指定來源（優先選擇讀取本地 CI 產生的 SARIF/JSON 報告檔案，風險較低）解析弱點數據。
        * 產出: GET /api/vulnerabilities API，能返回標準化的弱點清單 JSON。
    2. 🟧 前端 CVE：弱點檢視功能
        * 任務: 在 UI 中新增一個入口（如 Tab 或按鈕），點擊後彈出一個 Modal 或面板，呼叫後端 API 並以可勾選的清單形式展示弱點。
        * 產出: 使用者能夠在 UI 上看到詳細的弱點列表。
    3. 🟧 後端 Text-to-SQL：查詢驗證與執行預覽
        * 任務: 實現 SqlValidationService 和 SqlQueryService，提供 SQL 語法驗證和帶 LIMIT 的安全查詢執行功能。
        * 產出: POST /api/sql/validate 和 POST /api/sql/execute-preview 兩個可用的 API。
    4. 🟨 前端 Text-to-SQL：完整化體驗
        * 任務: 在前端整合數據預覽功能，在結果區新增「數據預覽」Tab，並顯示 SQL 驗證狀態。
        * 產出: 使用者不僅能看到 SQL，還能看到少量真實數據，極大提升信任感。
    5. 🟨 後端 CVE：請求修復的紀錄功能
        * 任務: 實現 POST /api/vulnerabilities/fix，接收前端提交的待修復弱點列表，並僅將其記錄在日誌或數據庫中，暫不執行任何實際修復操作。
        * 產出: 前後端就「提交修復」這個動作達成合約，流程被打通。


* 階段三：高級功能實現與整合
    1. 🟧 後端 Text-to-SQL：整合真實 AI 模型
        * 任務: 將 TextToSqlService 中寫死的 mock 邏輯替換為對真實 NLP 模型的呼叫。
        * 產出: 平台具備真正的自然語言轉 SQL 的能力。
    2. 🟨 後端 CVE：實現自動依賴更新
        * 任務: 實現 POST /api/vulnerabilities/fix 的核心邏輯，專注於風險較低的依賴版本更新。服務需要能解析 pom.xml 或
          package.json，修改版本號，並可能觸發 CI/CD 流程。
        * 產出: 平台具備自動修復依賴版本漏洞的核心能力。
    3. 🟩 前端體驗優化
        * 任務: 為兩個功能增加篩選、排序、搜尋等輔助功能。為耗時操作增加非同步進度顯示 (WebSocket 或輪詢)。
        * 產出: 更專業、更流暢的使用者體驗。
    4. 🟩 後端 CVE：整合 GitHub API 與變更預覽
        * 任務: 將弱點數據來源從讀取本地檔案升級為直接呼叫 GitHub API。在執行修復前，實現一個「變更預覽」的 API，返回將要修改的 diff 內容。
        * 產出: 更即時的數據來源和更安全的修復流程。

* 🟥 關鍵路徑驗證:
    * 階段一結束時: 必須能透過前端介面，成功從後端獲取並展示一個數據庫的結構樹。
    * 階段二結束時: 必須能透過前端介面，成功從後端獲取並展示一個（基於本地檔案的）弱點清單。
* 🟧 高優先級驗證:
    * 端到端測試 (Text-to-SQL): 輸入一個問題，系統能返回（mock 的）SQL，並且 UI 能正確展示。
    * 端到端測試 (CVE): 弱點清單能被正確加載並顯示在 UI 中。
* 🟨 中優先級驗證:
    * 功能測試 (Text-to-SQL): 提交一段 SQL，系統能返回其預覽數據。
    * 功能測試 (CVE): 提交一個依賴更新請求，後端能成功修改 pom.xml 檔案。
* 🟩 低優先級驗證:
    * 效能測試: 在弱點/數據表數量超過 1000 個時，UI 載入和響應時間應在可接受範圍內。
    * 使用者驗收測試 (UAT): 「變更預覽」功能是否清晰易懂。

5. 潛在挑戰與風險


* 🟥 核心依賴風險: 真實 NLP 模型的準確性和穩定性將直接決定 Text-to-SQL 功能的成敗。GitHub API 的變更和速率限制將直接影響 CVE 功能的穩定性。
    * 緩解措施: 初期採用 Mock/讀取本地檔案的方式解耦依賴，優先打通流程。對外部 API 呼叫做好完整的錯誤處理、重試和快取機制。
* 🟧 安全性風險: 自動修改程式碼和依賴是最高風險的操作。錯誤的修改可能導致整個專案癱瘓。
    * 緩解措施: 嚴格遵循從低風險功能（僅記錄 ->
      更新依賴）到高風險功能（修改原始碼）的開發路徑。強制引入「變更預覽」和手動確認環節。所有寫操作必須在獨立的 Git 分支上進行。
* 🟨 整合與使用者體驗風險: 兩個功能雖然強大，但如果生硬地整合在一個介面中，可能導致 UI 混亂，使用者不知所措。
    * 緩解措施: 在設計階段就投入資源進行高保真原型設計和使用者研究，確保 UI
      流程的直觀性和合理性。初期保持兩個功能的相對獨立性，後續再根據使用者回饋尋找深度整合的機會。
